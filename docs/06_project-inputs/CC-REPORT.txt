БЛОК 7. Экспертная позиция
7.1 Топ-10 что должно быть в handoff-спеке (по важности)
Routing-контекст: имя роута, путь, параметры, какой shell, родитель/дети, redirect-fallback при 401/403.
Endpoint-карта: список вызовов API в порядке (GET для рендера, POST/PATCH для действий), с пагинацией, фильтрами, query-params, типом ответа.
Состояния (state matrix): idle | loading | empty | error | success + что показать в каждом. С error — какой code → какая reaction (toast/inline/redirect).
Зависимости от sores: какие сторы читает (read), какие мутирует (write), какие реактивно подписаны (watch).
Permission gate: реальная роль (user.role) vs viewMode; кто видит экран; кто видит конкретные действия на экране.
Бизнес-инварианты: что нельзя сделать (например «withdrawal только если balance >= 5000»), какой error-code прилетит, как его обработать UI'ем.
i18n-ключи: список ключей, которые надо завести в ru.json/en.json, с примерным русским текстом (но не как hardcode).
Композиция UI: какие компоненты UI Kit задействовать, в каком слое (shared/ vs ui/), какие новые завести.
Деньги/время/токены-формата: где formatMoney(cents, currency, locale, allowZero), где formatDuration, где formatDateShort с таймзоной (важно: всегда передавать practice.timezone, а не системное).
Acceptance: 5-7 проверок «экран готов когда…» — конкретные, тестируемые (можно черновик E2E-шагов в Gherkin).
7.2 Что архитектор пишет избыточно
Повторное объяснение Stack/TypeScript-strict — в каждом промпте есть «Vue 3 + <script setup lang="ts">, strict». Это уже в ARCHITECTURE.md §1, я читаю один раз.
Цели «зачем мы делаем этот экран» с продуктовой мотивацией — для генерации не нужно; нужны только инварианты («что должно произойти»), а почему — отдельный документ.
Полное цитирование OpenAPI-схемы в промпте — если она в generated.ts, достаточно сказать «использует PaginatedPracticesResponse».
Подробное описание уже готовых компонентов (VButton, VInput) — я читаю их defineProps напрямую.
7.3 Что архитектор пропускает (узнаю постфактум)
Какие i18n-ключи уже существуют vs какие надо создать (часто пишу хардкод, потом возврат).
Что показать на 422 валидации vs 400 бизнес-ошибке — обычно решаю наугад, потом замена на toast/inline по фидбэку.
Какой timezone использовать в формате — Europe/Moscow? practice.timezone? Браузерный? Часто пропускается.
Что делать при пустой коллекции — текст empty-state почти никогда не предзадан.
Что делать с deep-link параметрами (open_practice__{uuid} через platform.getStartParam()) — это redirect где? до auth или после?
Что считать «owner action» vs «public action» на странице (например, кнопка «изменить» на PracticeDetailView для мастера-автора).
7.4 Формат спеки
Гибрид: декларация + чек-лист в одной структуре.

Декларация — для контракта (роут, endpoints, типы, permission, состояния) — её можно сверять автоматически.
Чек-лист — для acceptance + edge-cases — он читается линейно перед генерацией.
Story-форматы (BDD) удобны для acceptance, но плохи для declarative-полей (типы, endpoints) — там излишний glue-text.
7.5 Объём
< 1 страницы — недостаточно для нетривиального экрана (Master Apply, Withdrawal Approve).
1–3 страницы — sweet spot для типового CRUD-экрана с 1-3 endpoints.
5+ страниц — стена, нужен ToC + back-references. Если spec выходит за 5 страниц — экран нужно разбить на под-страницы или вынести инварианты в общую HANDOFF-VELO-PROFILE.md.

VELO-специфика для HANDOFF-VELO-PROFILE.md (не влезает в универсал):
