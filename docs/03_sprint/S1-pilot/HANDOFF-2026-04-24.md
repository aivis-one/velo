## 4. Про мердж в `main`

Честный ответ: **технически — можно прямо сейчас, процедурно — лучше после C05**.

### Почему можно:

- Все гейты зелёные (typecheck, lint errors = 0, build).
- Бэкенд не задет — API-контракт (`frontend/src/api/types.ts`) не менялся.
- `main` и `new_desing` расходятся в разных файлах: `main` впереди на 2 коммита (docker-compose.yml + frontend/Dockerfile — d398c78, 3564b82), `new_desing` впереди на 4 коммита (фронт + docs). Конфликтов при мердже быть не должно.
- Работать на одной ветке для сингл-дев проекта — здравый паттерн.

### Почему осторожно:

- C05 пока **не пройден**. Если на staging тени поедут — откатывать один коммит с 16k строк неудобно. Правильный порядок: деплой → C05 manual check → если ок, мердж в main.
- По плану спринтов мердж в main заложен на S4+ ([S2-SPRINT.md:53](../S2-bundle-port/S2-SPRINT.md)), но это изначально consesrvative-план, не хард-требование.

### Мой рекомендованный порядок:

1. **Сейчас:** задеплоить `new_desing` на staging.
2. **Артём + бэкендер:** пройтись глазами по экранам, убедиться что ничего визуально не сломано — в обеих темах (light сейчас дефолт, dark можно включить через DevTools `[data-theme="dark"]` на `<html>`).
3. **Если ок:** смержить `new_desing` → `main`, дальше Артём пилит циклы прямо в `main`, без отдельной ветки.
4. **Если что-то сломано:** фиксим на `new_desing`, мерджим после C05 green.

### Если решите мержить прямо сейчас:

```bash
git checkout main
git pull
git merge new_desing     # ожидается fast-forward или тривиальный merge
git push
```

Плюс один нюанс: в `main` два коммита с одинаковым сообщением `refactor(frontend): close tech debt blocks 1-2` (d398c78 + 3564b82) — это дубликат, наверное косяк прошлого мерджа. На работу не влияет, просто обратите внимание если будете чистить историю.

## 10. Оценка работы — честно

Артём — не программист, всё делается через Claude Code по протоколу SPEC v3.2-velo.

**Что объективно хорошо:**
- Дизайн-бандл зафиксирован как SSOT — в будущем не будет «откуда взялся этот цвет»
- Убраны `backdrop-filter` и glassmorphism — правильное решение для Telegram WebView (перформанс)
- Админские вьюхи сознательно отложены — нет распыления
- Зелёные гейты, нет техдолга в инфраструктуре
- Ведётся decisions-лог: каждое архитектурное решение имеет письменную мотивацию

**Что стоит иметь в виду:**
- Визуально экраны **пока не портированы**, сделана только инфра. Не жди новых экранов на staging — ты увидишь текущие экраны на новых токенах. Порт первых двух экранов (Dashboard + Welcome) запланирован на C10–C11.
- Работа идёт через Claude Code, значит правки иногда будут выглядеть избыточно структурированно — это часть протокола, не излишество
